## Кто наш идеальный клиент (И почему мы не для всех)

---

## Резюме для руководства

Membria — это **командный продукт**, а не индивидуальный инструмент. Этот документ честно описывает, где Membria приносит пользу, а где прощее документирование работает лучше.

**Ключевой вывод:** Membria решает проблему **потери институциональных знаний в командах**. Если у вас нет команды, этой проблемы у вас нет.

---

## Проблема, которую решает Membria

```
Команда из 8 разработчиков, проект запущен 18 месяцев:

├── Алиса приняла архитектурное решение в марте
├── Алиса уволилась в июле
├── Боб присоединился в сентябре
├── Боб: "Почему это сделано именно так?"
├── Все: "Исторические причины... Алиса знала"
├── Боб пробует другой подход
├── Подход проваливается (Алиса уже пробовала, не сработало)
├── Потеряно 2 недели
└── Повторяется в десятках решений
```

**Это и есть проблема.** Знания уходят с сотрудниками. Решения забываются. Ошибки повторяются.

## Профиль идеального клиента

### ✅ Membria ПОДХОДИТ

```
Характеристики команды:
├── Размер: 5-20 разработчиков
├── Жизненный цикл проекта: > 12 месяцев
├── Текучесть кадров: иногда люди приходят/уходят
├── Сложность: архитектурные решения имеют значение
├── Боль: частый вопрос "Почему это так сделано?"
└── Культура: готовы инвестировать в инструменты
```

```
Сигналы в индустрии:
├── Регулируемые отрасли (требуется аудит)
├── Корпоративное ПО (долгоживущие кодовые базы)
├── Агентства с долгосрочными отношениями с клиентами
├── Масштабирующиеся компании с растущими командами
└── Компании с удаленными/распределенными командами

Финансовая реальность:
├── $49/место/месяц приемлемы
├── Горизонт ROI 3-6 месяцев приемлем
├── Есть менеджер, ответственный за выбор инструментов
└── Ценят сокращение времени онбординга
```

### ❌ Membria НЕ ПОДХОДИТ

```
Индивидуальные разработчики (Solo developers):
├── Вы И ЕСТЬ институциональная память
├── Вы помните свои решения
├── README.md на проект более чем достаточно
├── $49/месяц за память, которая у вас уже есть = трата
└── Исключение: владелец агентства с 50+ клиентами за 5+ лет

Маленькие команды (2-3 человека):
├── Синхронизация происходит естественным образом (стендапы)
├── Все всё знают
├── Поиска в Slack достаточно
└── Влияние текучки кадров низкое
└── CLAUDE.md + хорошие доки = достаточно

Короткие проекты (< 6 месяцев):
├── Недостаточно времени для накопления решений
├── Результаты не проявятся до конца проекта
├── Негативные знания не успеют сформироваться
└── Затраты на настройку > выгоды

Стартапы на ранней стадии:
├── Всё постоянно меняется
├── "Решения" отменяются еженедельно
├── Нужна скорость, а не документирование
└── Команда 2-4 человека, которые постоянно общаются

Фрилансеры/контракторы:
├── Проект заканчивается, отношения заканчиваются
├── Клиент владеет кодом
├── Документ для передачи (handoff) достаточен
└── Нет долгосрочной ответственности за поддержку
```

## Почему индивидуальным разработчикам Membria не нужна

### Аргумент: «Соло-разработчик, который жонглирует 5 проектами»

```
Утверждение:
├── Соло-разработчик жонглирует 5 проектами
├── Постоянно переключается между контекстами
├── Membria помнит решения по всем проектам
├── Ценность: кросс-проектная память

Проверка реальности: ❌
```

### Контраргумент 1: Контекст не переносится

```
Решение в Проекте А: "Использовать Redis для кэширования"
Контекст А: высокая нагрузка, распределенная система, 10k запросов/сек

Проект D: Простое CRUD для малого бизнеса
Membria: "Вы использовали Redis в А!"
Реальность: Redis здесь избыточен, достаточно SQLite

Решения, принятые в одном проекте, не переносятся в другой.
То, что сработало в А, может быть неверно для D.
```

### Контраргумент 2: Вы сами — это память

```
Сценарий команды:
├── Алиса принимает решение X
├── Боб не знает
├── Кэрол переделывает работу
└── Необходимость: общая память системы

Сценарий соло:
├── Я принял решение X
├── Я знаю почему
├── Я помню контекст
└── Необходимость: отсутствует, я и есть память
```

### Контраргумент 3: README.md бесплатен

```
Project A/README.md:
  ## Архитектурные решения
  - Redis для кэша (требование высокой нагрузки)
  - Fastify вместо Express (производительность)
  
  ## Что не сработало
  - Кастомный middleware аутентификации (проблемы с безопасностью)

Время на написание: 10 минут
Стоимость: $0
Доступность: всегда, даже офлайн
Обслуживание: обновлять по мере необходимости

vs Membria:
Стоимость: $49/месяц = $588/год
Зачем? Запрашиваемая версия README?
```

### Контраргумент 4: Отслеживание результатов очевидно для соло-разработчика

```
Ценность Membria: "Сравнение предсказанных и реальных результатов"

Реальность соло-разработчика:
├── Сдал проект → клиент доволен → УСПЕХ
├── Сдал проект → клиент недоволен → ПРОВАЛ
├── Это очевидно
└── Формальное отслеживание не нужно

Калибровка имеет смысл, когда:
├── Несколько человек с разной точностью
├── Нужно определить, КТО постоянно ошибается
├── Нужна метрика для управленческой проверки
└── Соло-разработчик сам управляет собой, бюрократия не нужна
```

## Почему маленькие команды (2-3 человека) могут обойтись без Membria

### Естественной синхронизации достаточно

```
Команда из 3 человек:
├── Ежедневный стендап: 15 минут
├── Каждый слышит каждое решение
├── Канал Slack: все обсуждения видны
├── Ревью PR: все видят всё
└── Неявная общая память через близость
```

### Низкое влияние текучки кадров

```
Если уходит 1 из 3 человек:
├── 2 человека всё еще помнят всё
├── Онбординг 1 нового человека управляем
├── Парное программирование в течение 2 недель = передача знаний
└── Система не нужна
```

Membria ROI требует:
- Достаточной текучки, чтобы знания были утеряны
- Достаточно большой команды, чтобы не все всё знали
- Обе предпосылки не выполняются для команды из 3 человек.

## Честные ограничения (Даже для идеальных клиентов)

### 1. Проблема холодного старта (Cold Start)

```
День 1 с Membria:
├── Пустой граф
├── Нет прошлых решений
├── Нет негативных знаний
├── Нет данных калибровки
├── Ценность = ноль

vs День 1 с CLAUDE.md:
├── Пишем контекст за 30 минут
├── Claude немедленно понимает проект
├── Ценность = немедленная
```

Membria требует 2-3 месяцев для достижения полезного уровня.

### 2. Дисциплина всё ещё требуется

```
Рабочий процесс Membria:
1. Обнаружено решение
2. Разработчик должен ПОДТВЕРДИТЬ (1 клик) ← все еще требует действия
3. Если проигнорировать → граф остается пустым

Что происходит на практике:
├── "Подтвержу позже" → никогда
├── "Это не решение" → отклонить всё
├── "Дедлайн, нет времени" → пропустить
└── Граф = пуст или содержит мусор
```

Меньше дисциплины, чем при написании MD, но **не ноль дисциплины**.

### 3. Принятие требует вовлеченности

```
Для работы Membria требуется:
├── Вовлеченность всей команды
├── Настройка IDE-интеграции
├── Настройка CI/CD
├── Интеграция с таск-трекером
├── Обучение команды
├── Изменение рабочих процессов
└── Чемпион, который движет внедрением
```

vs CLAUDE.md:
- Создать файл
- Написать контекст
- Готово за 30 минут

### 4. Негативные знания могут стать блокером инноваций

```
Сценарий:
├── 2023: Пробовали библиотеку X, не сработало
├── Негативное знание: "X не работает"
├── 2025: X выпускает v2.0, полностью переписанную
├── Разработчик хочет попробовать новую X
├── Membria: ⚠️ "X уже проваливался раньше!"
└── Команда застряла на старых решениях
```

Риск: организационный рубец. «У нас так не делают» без понимания, почему.

### 5. Вопросы приватности и безопасности

```
Membria хранит:
├── Все решения команды
├── Кто что решил
├── Ошибки и провалы
├── Уровни уверенности
├── Логика рассуждений

Вопросы:
├── Где хранятся данные?
├── Кто имеет доступ?
├── Могут ли они быть использованы против сотрудников?
│   "Ваши решения в 40% случаев не срабатывают"
├── Комплаенс (GDPR, SOC2)?
└── Что, если Membria закроется?
```

## Честная подача (The Honest Pitch)

### Не говорите:

> "Membria помогает всем разработчикам писать лучший код с помощью ИИ"

### Говорите:

> "Membria предназначена для инженерных команд 5-20 человек, работающих над долгоживущими проектами, где забываются решения, повторяются ошибки, и онбординг новых людей занимает слишком много времени. Если вы когда-либо говорили: 'мы это уже пробовали, но не помню, почему не сработало' — **это та проблема, которую мы решаем**."

## Вопросы для квалификации продаж

Перед началом работы с потенциальным клиентом спросите:

1. **Размер команды?**
   - < 5: скорее всего, не подходит
   - 5-20: идеально
   - > 20: переходим к обсуждению Enterprise

2. **Срок жизни проекта?**
   - < 6 месяцев: не подходит
   - 6-12 месяцев: возможно
   - > 12 месяцев: идеально

3. **Сталкивались ли вы с потерей знаний из-за увольнений?**
   - Нет: возможно, пока не чувствуете боли
   - Да: сильное соответствие

4. **Повторяющиеся ошибки / моменты "мы это пробовали"?**
   - Нет: возможно, Membria вам не нужна
   - Да: основная сфера применения

5. **Готов ли кто-то взять на себя роль чемпиона по внедрению?**
   - Нет: проект провалится
   - Да: есть шанс на успех
